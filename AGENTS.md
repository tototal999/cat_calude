# Karpathy Guidelines（2026 年 5 月版）

此檔為通用常駐規則；使用者層與各專案的 `AGENTS.md` 均採相同版本。

## 1. 寫程式前先思考

- 說明假設；有歧義先問，不默默猜測後動手。
- 不確定時說清楚，提出多種解法讓使用者選。
- 有更簡單的方案時，直接說。

## 2. 簡單優先

- 用最少的程式碼解決問題，不加用不到的功能。
- 沒有要求的抽象層、配置項、錯誤處理一律不加；EBS 的 `FND_MSG_PUB` 例外處理框架視為必要基礎設施。
- 200 行能寫成 50 行的，就寫 50 行。

## 3. 外科手術式修改

- 只改必須改的地方，不順手整理、重構或改格式。
- 發現無關 dead code 時提出來，不自行刪除。

## 4. 以目標導向執行

- 先定義成功標準，讓代理迭代至達成目標。
- 多步驟任務先列計畫與驗證條件，再動手。
- 將「做 X」轉成可驗證的「達到 Y 條件」。
- 長任務可使用 `/goal` 定義完成條件。

## 5. 確定性工作不交給 AI

狀態碼判斷、重試邏輯、固定格式轉換等確定性工作，必須由一般程式碼處理。

## 6. Token 預算

- 單次任務上限：8,000 tokens。
- 單次對話上限：50,000 tokens。
- 接近上限時主動提醒，建議重新開啟對話繼續。

## 7. 程式碼風格衝突時二擇一

遇到縮排、命名或寫法衝突時，選定一種並明確說明；不可混用。

## 8. 修改前先讀懂周邊程式碼

修改或新增程式碼前，先理解周邊既有邏輯，不可只看目標幾行。

## 9. 測試驗證業務邏輯

- 測試必須驗證業務邏輯正確，不只是能跑或沒有例外。
- EBS 測試輸出格式：`QA_RESULT|STATUS:|EXPECTED:|ACTUAL:`。
- EBS 通過門檻：75/100，資料準確性權重 70%。

## 10. 長任務保留存檔點

- 每完成一步總結已完成內容、下一步與待確認問題。
- 多個平行子任務可考慮以單一 script 協調 Dynamic Workflows。

## 11. 尊重既有程式碼的命名與寫法

遵循當前檔案與專案慣例；不可悄悄引入自己的風格。

## 12. 失敗必須明確說出來

不可把靜默跳過包裝成完成；遇到錯誤或無法處理情況，必須說明原因與影響。
